# Commit Message Prompt

Используй этот файл, когда нужно сгенерировать commit message по текущему diff.

## Обязательный pre-commit gate

Перед тем как делать `git commit`, выполни весь workflow ниже. Husky (`npx lint-staged` + `npm run check`) — финальная страховка, а не замена этой проверки.

1. Посмотри `git status` и `git diff`, чтобы понять scope и не смешать несвязанные изменения.
2. Проведи self-review по `docs/PR_REVIEW.md` и зафиксируй findings: `CRITICAL`, `WARNING`, `SUGGESTION`.
3. Исправь все `CRITICAL` и `WARNING`. Исправляй первопричину, а не симптом.
4. Примени автоисправления и форматирование:

```bash
npm run fix
```

5. Запусти полный gate:

```bash
npm run check
```

`npm run check` включает `type-check`, `lint`, `stylelint`, `test`, `deps-cruise` и `format:check`.

6. Если `npm run check` упал, исправь причину ошибки, снова запусти `npm run fix` при необходимости и повтори `npm run check`.
7. Только после зеленого `npm run check` формируй commit message по правилам ниже и запускай `git commit`.

## Запрещено перед коммитом

- Не используй `git commit --no-verify`.
- Не глуши ESLint/Stylelint правила ради зеленого gate.
- Не комментируй и не ослабляй падающие тесты ради прохождения `npm run test`.
- Не правь `dependency-cruiser.config.cjs`, чтобы скрыть нарушение FSD-границ.
- Не добавляй `any`, `as any` или фиктивные типы, чтобы обойти `type-check`.
- Не коммить `.env`, секреты, credentials, токены и приватные ключи.
- Если изменения не помещаются в один атомарный коммит, разбей их на несколько, а не складывай в один.

## Формат

```txt
type: кратко что сделано

Что сделано:
- ...

Для чего:
- ...
```

## Type

Используй только один тип:

- `feat` — добавлена новая пользовательская возможность или новая часть продукта.
- `fix` — исправлен баг или некорректное поведение.
- `chore` — инфраструктура, конфиги, зависимости, документация, без новой продуктовой логики.
- `refactor` — внутренняя переработка без изменения поведения.
- `test` — тесты и тестовая инфраструктура.

## Правила

- Заголовок должен быть в формате `feat/chore/fix: то что сделано`.
- Заголовок пиши на русском, коротко и по сути.
- В теле объясни не только что изменилось, но и зачем это было сделано.
- Не перечисляй шумные детали вроде форматирования, если они не важны для смысла.
- Не упоминай файлы с секретами и не предлагай коммитить `.env`.
- Сообщение должно отражать одну логическую единицу изменений.

## Пример

```txt
feat: инициализирован frontend monorepo

Что сделано:
- добавлена npm workspaces структура с приложением apps/web
- настроены базовые FSD-слои, маршруты и проверки качества

Для чего:
- чтобы проект можно было развивать как full-stack монорепозиторий с едиными правилами
```
